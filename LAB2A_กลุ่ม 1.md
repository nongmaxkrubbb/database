## 🟦 Section 1 — Relational Model (ตาราง)

**1.1 ข้อมูลร้านกาแฟถูกแบ่งเป็นกี่ตาราง และทำไมต้องแยก ORDER_ITEM ออกจาก ORDER**

ข้อมูลถูกแบ่งออกเป็น 5 ตาราง คือ MENU, INGREDIENT, MEMBER, ORDER และ ORDER_ITEM ต้องแยก ORDER_ITEM ออกจาก ORDER เพราะคำสั่งซื้อ 1 รายการ (1 order) สามารถมีเมนูได้หลายอย่างพร้อมกัน (เช่น O001 สั่งทั้งอเมริกาโน่และลาเต้) ถ้ารวมไว้ในตาราง ORDER ตารางเดียว แต่ละแถวจะต้องมีหลายคอลัมน์ซ้ำ ๆ กัน (menu1, qty1, menu2, qty2, ...) ซึ่งไม่รู้ล่วงหน้าว่าจะต้องเผื่อกี่คอลัมน์ และจะทำให้เกิด Repeating Group ซึ่งขัดกับหลักการ Normalization (1NF) การแยกตาราง ORDER_ITEM ออกมาทำให้แต่ละแถวแทน 1 รายการสินค้าใน 1 order เท่านั้น รองรับจำนวนรายการที่ไม่จำกัด และเชื่อมกลับไปยัง ORDER และ MENU ด้วย Foreign Key (order_id, menu_id) ได้อย่างเป็นระเบียบ

**1.2 (ก) แก้ราคาลาเต้ใน Relational — ต้องแก้ที่ตารางไหน กี่แถว**

แก้ที่ตาราง MENU เพียงตารางเดียว และแก้เพียง 1 แถว คือแถวที่ menu_id = M002 เปลี่ยนค่า price จาก 75 เป็น 80 ตารางอื่น (ORDER_ITEM, ORDER) ไม่ต้องแก้ เพราะไม่ได้เก็บราคาซ้ำไว้ — เวลาต้องใช้ราคาก็ไป JOIN ดึงจากตาราง MENU เสมอ

**1.2 (ข) ถ้าเป็น Excel ไฟล์เดียว**

ราคาลาเต้จะถูกเขียนซ้ำในทุกแถวที่มีการสั่งลาเต้ เช่น ถ้ามีคนสั่งลาเต้ไป 500 ครั้ง ก็ต้องไล่แก้ราคาทีละแถวถึง 500 แถว และถ้าแก้ไม่ครบทุกแถว ข้อมูลราคาลาเต้ในระบบจะไม่ตรงกัน (Update Anomaly) เกิดความไม่สอดคล้องของข้อมูล (Data Inconsistency)

**1.2 (ค) ข้อดีของ Relational คือ**

มี "แหล่งความจริงเดียว" (Single Source of Truth) — ราคาลาเต้เก็บอยู่ที่เดียวคือแถว M002 ในตาราง MENU เมื่อแก้ครั้งเดียวก็ถูกต้องทั้งระบบทันที ลดความซ้ำซ้อนของข้อมูล (Redundancy) และลดโอกาสเกิดข้อมูลขัดแย้งกัน ตามหลักการ Normalization

**1.3 Primary Key คือ**

คอลัมน์ (หรือกลุ่มคอลัมน์) ที่ใช้ระบุแต่ละแถวในตารางให้ไม่ซ้ำกัน (Unique) และห้ามเป็นค่าว่าง (Not Null) เปรียบเหมือนเลขบัตรประชาชนของแต่ละแถว

*เหตุผลที่ menu_id และ member_id เป็น Primary Key:* menu_id เป็น PK ของตาราง MENU เพราะแต่ละเมนูต้องมีรหัสไม่ซ้ำกัน แม้ชื่อเมนูอาจซ้ำกันได้ เช่นเดียวกัน member_id เป็น PK ของตาราง MEMBER เพราะชื่อสมาชิกอาจซ้ำกันได้แต่รหัสต้องไม่ซ้ำ และรหัสเหล่านี้ยังถูกใช้เป็น Foreign Key ในตารางอื่นเพื่อเชื่อมความสัมพันธ์ระหว่างตารางด้วย

**1.4 ❓ สิ่งที่ยังสงสัย**

ถ้าวัตถุดิบ 1 ชนิดถูกใช้ในหลายเมนู (เช่น นม ใช้ทั้งในลาเต้และช็อกโกแลต) ความสัมพันธ์ระหว่าง MENU กับ INGREDIENT จะกลายเป็นแบบ Many-to-Many หรือไม่ และถ้าใช่ ควรต้องมีตารางกลาง (Junction Table) เพิ่มอีกหรือเปล่า?

## 🟨 Section 2 — Key-Value Database

**2.1 GET menu:M001:price ต้องรู้อะไรล่วงหน้า**

ต้องรู้ Key ที่แน่นอนล่วงหน้า คือต้องพิมพ์ `menu:M001:price` ให้ตรงเป๊ะทั้งชื่อประเภท รหัส และชื่อ field ถ้าไม่รู้รหัส M001 มาก่อน ก็ไม่สามารถดึงข้อมูลได้เลย เพราะ Key-Value Database ไม่มีการค้นหาด้วยเงื่อนไข

*สิ่งที่ Relational ทำได้แต่ Key-Value ทำไม่ได้:* Relational สามารถ Query แบบมีเงื่อนไข (WHERE) ได้ เช่น `SELECT * FROM MENU WHERE price < 65` โดยไม่ต้องรู้ล่วงหน้าว่ามีเมนูอะไรบ้าง ในขณะที่ Key-Value ทำแบบนี้ไม่ได้โดยตรง เพราะถูกออกแบบมาเพื่อความเร็วในการดึงข้อมูลเมื่อรู้ Key ที่แน่นอนเท่านั้น

**2.2 Key-Value Command สำหรับเก็บวัตถุดิบของอเมริกาโน่**

```
SET menu:M001:ingredients "กาแฟ,น้ำร้อน"
```
หรือถ้ารองรับ List: `RPUSH menu:M001:ingredients "กาแฟ" "น้ำร้อน"` แล้วดึงด้วย `LRANGE menu:M001:ingredients 0 -1`

**2.3 3 กรณีที่ AI บอกว่า Key-Value เหมาะกับ e-commerce**

(1) เก็บ Session ของผู้ใช้ที่ล็อกอินอยู่ เพื่อให้เว็บตรวจสอบสถานะล็อกอินได้เร็ว (2) เก็บตะกร้าสินค้า (Shopping Cart) ชั่วคราวระหว่างที่ลูกค้ากำลังช้อปปิ้ง (3) ใช้เป็น Cache สำหรับข้อมูลที่ถูกเรียกดูบ่อย เช่น จำนวนสต็อกสินค้าล่าสุด

*ข้อมูลร้านกาแฟที่น่าใช้ Key-Value มากที่สุด:* ราคาปัจจุบันของแต่ละเมนู เพราะต้องถูกอ่านบ่อยที่สุดและต้องการความเร็วสูงสุด

**2.4 ❓ สิ่งที่ยังสงสัย**

ถ้าข้อมูลใน Key-Value ไม่มีความสัมพันธ์ระหว่างกันเลย แล้วเวลาต้องออกรายงานสรุปยอดขายทั้งร้าน (ซึ่งต้องรวมข้อมูลจากหลาย Key) ต้องดึงมาประมวลผลที่ฝั่งโปรแกรม (Application) เองใช่หรือไม่?

## 🟩 Section 3 — Document-Oriented Database

**3.1 Document (รวมวัตถุดิบไว้ใน Document เดียว) vs Relational (แยกตาราง)**

| | ข้อดี | ข้อเสีย |
|---|---|---|
| **Document** | อ่านข้อมูลเมนู 1 รายการได้ครบในครั้งเดียว ไม่ต้อง JOIN เหมาะกับกรณีที่มักอ่านข้อมูลทั้งชุดพร้อมกัน | ถ้าวัตถุดิบเดียวกันถูกใช้หลายเมนู ข้อมูลจะถูกเก็บซ้ำในหลาย Document แก้ไขต้องไล่แก้ทุกที่ |
| **Relational** | วัตถุดิบแต่ละชนิดถูกเก็บเป็นข้อมูลอ้างอิงเดียว ไม่ซ้ำซ้อน แก้ที่เดียวมีผลทุกเมนูที่ใช้ | ต้องใช้ JOIN ทุกครั้งที่ต้องการดูวัตถุดิบของเมนู ทำให้ Query ซับซ้อนขึ้น |

**3.2 field ที่ไม่เท่ากัน (เช่น seasonal_note)**

*ถ้าเป็น Relational:* ต้องเพิ่มคอลัมน์ seasonal_note ในตาราง MENU ให้ทุกแถว แต่แถวที่ไม่มีข้อมูลนี้ต้องเก็บเป็นค่า NULL เพราะ Schema กำหนดตายตัวว่าทุกแถวต้องมีคอลัมน์ชุดเดียวกัน

*ข้อได้เปรียบของ Document:* มี Schema ที่ยืดหยุ่น (Flexible Schema) แต่ละ Document ไม่จำเป็นต้องมี field ชุดเดียวกัน จึงเพิ่ม field พิเศษให้เฉพาะบาง Document ได้ทันทีโดยไม่กระทบ Document อื่น

**3.3 ถ้าเปลี่ยนชื่ออเมริกาโน่เป็น "Black Coffee"**

ต้องแก้ชื่อใน Document ของ collection menus (M001) และต้องไปแก้ชื่อที่ซ้ำอยู่ใน items ของ Document คำสั่งซื้อ O001 ด้วย ถ้าลืมแก้จุดใดจุดหนึ่ง ข้อมูลจะไม่ตรงกัน — ออร์เดอร์เก่าจะยังโชว์ชื่อเก่าทั้งที่เมนูปัจจุบันเปลี่ยนไปแล้ว

*เชื่อมกับปัญหาบทที่ 1:* นี่คือปัญหา Data Redundancy ที่นำไปสู่ Update Anomaly เช่นเดียวกับที่เรียนในบทที่ 1 อย่างไรก็ตาม Document Database มักยอมรับการเก็บซ้ำแบบนี้โดยตั้งใจ (Denormalization) เพื่อแลกกับความเร็วในการอ่าน โดยฝั่งแอปพลิเคชันต้องรับผิดชอบดูแลความสอดคล้องของข้อมูลเอง

**3.4 ❓ สิ่งที่ยังสงสัย**

มีวิธีใดที่ Document Database จะช่วยลดปัญหาข้อมูลซ้ำซ้อนแบบนี้ได้บ้าง เช่น การอ้างอิง (Reference) แทนการฝัง (Embed) ข้อมูล และควรเลือกใช้แบบไหนเมื่อไหร่?

## 🟪 Section 4 — Graph Database

**4.1 โหนดและ Edge**

*โหนด:* สมาชิก (MEMBER เช่น สมชาย, มาลี) และเมนู (MENU เช่น อเมริกาโน่, ลาเต้, ช็อกโกแลต)

*Edge:* ความสัมพันธ์ระหว่างโหนด เช่น ORDERED (ใครสั่งเมนูอะไร), LIKES (ใครชอบเมนูอะไร), REFERRED (ใครแนะนำใครมาสมัครสมาชิก)

*ตัวอย่างชีวิตจริงที่เป็น Graph:* ระบบเพื่อนใน Facebook, ระบบแผนที่/เส้นทางถนนใน Google Maps, ระบบแนะนำหนังใน Netflix

**4.2 "ลูกค้าที่ชอบลาเต้ มักสั่งอะไรอีก?" — Graph vs Relational**

*ทำใน Graph:* ใช้การไล่ตาม Edge จากโหนดหนึ่งไปอีกโหนดหนึ่งโดยตรง (Traversal) ทำได้เร็วมากไม่ว่าจะกี่ขั้นความสัมพันธ์

*ทำใน Relational:* ต้องเขียน Query ที่ JOIN ตารางหลายต่อ (MEMBER JOIN ORDER_ITEM JOIN MENU ซ้ำหลายรอบ) ยิ่งความสัมพันธ์ซับซ้อนหรือหลายขั้น (multi-hop) ก็ยิ่งต้อง JOIN มากขึ้น

*Graph เหมาะกว่าเพราะ:* คำถามประเภทนี้เน้นความสัมพันธ์หลายขั้น (multi-hop relationship) ซึ่งเป็นจุดแข็งโดยตรงของ Graph Database

**4.3 แอป Graph Database ในชีวิตประจำวัน**

(1) LinkedIn/Facebook — หาความเชื่อมโยงระหว่างคน เช่น "เพื่อนของเพื่อน" (2) Google Maps — หาเส้นทางสั้นที่สุดโดยเดินตามโครงข่ายถนน (3) Netflix/Spotify — ระบบแนะนำเนื้อหาจากความสัมพันธ์ระหว่างผู้ใช้ที่ชอบคล้ายกัน

*แอปที่น่าสนใจที่สุด:* Google Maps เพราะแสดงให้เห็นชัดว่า Graph เหมาะกับปัญหาการหาเส้นทาง (Path Finding) ซึ่ง Relational ทำได้ยากและช้ามากเมื่อข้อมูลมีจำนวนโหนดและ Edge มหาศาล

**4.4 ❓ สิ่งที่ยังสงสัย**

Graph Database เก็บข้อมูลจำนวนมาก (เช่น สมาชิกหลักล้านคน) ได้มีประสิทธิภาพเท่ากับ Relational หรือไม่ หรือมีข้อจำกัดด้าน Scalability ที่ต้องระวังเป็นพิเศษ?

## 🔄 Section 5 — มารวมกัน: เปรียบเทียบและสรุป

### 5.1 ตารางเปรียบเทียบ

| หัวข้อ | Relational | Key-Value | Document | Graph |
|---|---|---|---|---|
| หน่วยจัดเก็บ | ตาราง (Table) — แถวและคอลัมน์ | คู่ Key-Value | Document (JSON) | โหนดและ Edge |
| ดึงข้อมูลโดย | Query เงื่อนไข (WHERE) | รู้ Key ที่แน่นอน | Query บน field ใน Document | เดินตามความสัมพันธ์ (Traversal) |
| วัตถุดิบเมนู | แยกตาราง INGREDIENT | เก็บเป็น string/list ในค่า | ฝังเป็น array ใน Document เดียวกัน | เป็นโหนด/Edge แยก (ถ้าต้องการ) |
| เปลี่ยนราคา 1 รายการ | แก้ 1 แถวใน MENU | แก้ 1 Key | แก้ field price ใน 1 Document | แก้ property price ของ 1 โหนด |
| DBMS ตัวอย่าง | MySQL, PostgreSQL | Redis | MongoDB | Neo4j |
| เหมาะกับ | ข้อมูลมีโครงสร้างแน่นอน ต้องการความถูกต้องสูง | ข้อมูลอ่าน/เขียนเร็วมาก รู้ Key ล่วงหน้า | ข้อมูลโครงสร้างยืดหยุ่น อ่านเป็นก้อนเดียว | ข้อมูลที่ความสัมพันธ์สำคัญ |

### 5.2 สำหรับร้านกาแฟ "ดอยเด่น" — ควรใช้รูปแบบไหน

| ข้อมูล | รูปแบบที่เหมาะสม | เหตุผล |
|---|---|---|
| เมนูและวัตถุดิบ | Relational หรือ Document | โครงสร้างค่อนข้างคงที่ Relational ช่วยลดความซ้ำซ้อนของวัตถุดิบที่ใช้ร่วมกันหลายเมนู แต่ถ้าต้องการความยืดหยุ่น Document ก็เหมาะเช่นกัน |
| ข้อมูลสมาชิก + แต้มสะสม | Relational | โครงสร้างชัดเจน ต้องการความถูกต้องแม่นยำ และอาจต้องเชื่อมกับตารางอื่น |
| ราคาเมนู Real-time (เปลี่ยนบ่อย) | Key-Value | ต้องการความเร็วสูงสุด และมักรู้ Key (menu_id) อยู่แล้ว เหมาะเป็น Cache |
| คำสั่งซื้อ (ต้องย้อนดูประวัติ) | Relational หรือ Document | ต้องการความถูกต้องเชิงธุรกรรม Relational รายงานยอดขายได้ดี Document เหมาะถ้าต้องการอ่านเร็วแบบก้อนเดียว |
| ระบบแนะนำเมนู (ใครชอบอะไร) | Graph | เน้นความสัมพันธ์ระหว่างสมาชิกกับเมนู ซึ่งเป็นจุดแข็งของ Graph โดยเฉพาะ |

### 5.3 คำถามเปรียบเทียบเชิงลึก

**Q1 (ก) Schema คือ** โครงสร้างหรือ "พิมพ์เขียว" ของข้อมูล เช่น การกำหนดว่าตาราง MENU มีคอลัมน์อะไรบ้าง ในกรณีนี้ "โครงสร้างตาราง MENU" คือ Schema

**Q1 (ข) Instance คือ** ข้อมูลจริงที่ถูกเก็บอยู่ในโครงสร้างนั้น ณ ช่วงเวลาหนึ่ง เช่นแถว "M001, อเมริกาโน่, 60, ร้อน" คือ Instance ของ Schema ตาราง MENU

**Q1 (ค) ใน Document Database สิ่งที่เทียบกับ Schema คือ** รูปแบบ/โครงสร้าง field ที่ Document ส่วนใหญ่ในคอลเลกชันนั้นมักมีร่วมกัน แม้จะไม่ถูกบังคับตายตัวเหมือน Relational (บางระบบเรียกว่า Flexible/Implicit Schema)

**Q2 100 สาขา + Real-time stock — Centralized หรือ Distributed**

ควรใช้แบบ Distributed เพราะข้อมูลสต็อกของแต่ละสาขาต้องอัปเดตแบบ Real-time พร้อมกันหลายจุด ถ้าใช้ฐานข้อมูลศูนย์กลางเดียว ทุกสาขาต้องส่งคำขอไปที่เซิร์ฟเวอร์เดียวกันตลอดเวลา อาจเกิดคอขวด (Bottleneck) ความหน่วงสูง และถ้าระบบกลางล่ม ทุกสาขาจะหยุดทำงานพร้อมกัน แบบ Distributed ช่วยให้แต่ละภูมิภาคเข้าถึงข้อมูลได้เร็วขึ้นและระบบยังทำงานต่อได้แม้บางโหนดมีปัญหา แต่ต้องแลกกับความซับซ้อนในการทำให้ข้อมูลสอดคล้องกันระหว่างโหนด

**Q3 "วันนี้ขายเมนูไหนมากที่สุด" — ทำได้ง่ายหรือยาก**

| รูปแบบ | ง่าย/ยาก | เหตุผล |
|---|---|---|
| Relational | ง่าย | ใช้ GROUP BY และ SUM/COUNT บนตาราง ORDER_ITEM ได้คำตอบในคำสั่งเดียว |
| Key-Value | ยาก | ต้องรู้ Key ของทุกออร์เดอร์ล่วงหน้าแล้วไล่ดึงมานับเอง หรือออกแบบ Key พิเศษไว้ล่วงหน้า |
| Document | ปานกลาง | ทำได้ด้วย Aggregation (เช่น $group ใน MongoDB) แต่ต้องไล่เข้าไปในโครงสร้าง array (items) ก่อน |
| Graph | ยาก/ไม่เหมาะ | Graph เก่งเรื่องความสัมพันธ์ ไม่ได้ออกแบบมาเพื่อการรวมยอด/นับจำนวนเชิงสถิติจำนวนมาก |

### 5.4 สิ่งที่กลุ่มค้นพบที่ไม่ได้อยู่ในสไลด์

**สิ่งที่น่าแปลกใจ/น่าสนใจที่สุด:** ข้อมูลชุดเดียวกันสามารถถูกจัดเก็บได้ถูกต้องในทั้ง 4 รูปแบบ แต่แต่ละรูปแบบจะ "เก่ง" ในงานที่ต่างกันโดยสิ้นเชิง ไม่มีรูปแบบไหนดีที่สุดเสมอไป ขึ้นอยู่กับโจทย์จริง และระบบใหญ่ ๆ มักผสมใช้หลายรูปแบบร่วมกัน (Polyglot Persistence)

**คำถามที่กลุ่มยังสงสัยมากที่สุด:** ถ้าระบบใช้ทั้ง Relational และ Key-Value (cache ราคา) พร้อมกัน จะมั่นใจได้อย่างไรว่าราคาใน Cache จะไม่ "ไม่ตรงกัน" กับฐานข้อมูลหลักเมื่อมีการเปลี่ยนราคา ต้องมีกลไกอะไรมาช่วยซิงค์ข้อมูล?
