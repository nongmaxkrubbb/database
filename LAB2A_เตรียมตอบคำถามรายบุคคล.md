# เตรียมตอบคำถาม — "ทำอะไรในงานนี้" (เผื่ออาจารย์สุ่มถาม)

## ภานุพงษ์ บำรุงศรี — Relational

ผมรับผิดชอบโมเดล Relational ครับ ทำความเข้าใจว่าข้อมูลร้านกาแฟถูกแบ่งเป็น 5 ตารางยังไง แล้วตอบคำถามว่าทำไมต้องแยกตาราง ORDER_ITEM ออกจาก ORDER ทำไม menu_id กับ member_id ถึงเป็น Primary Key และถ้าราคาเปลี่ยนต้องแก้ตรงไหน ผมสรุปได้ว่า Relational ดีตรงที่ข้อมูลไม่ซ้ำซ้อน แก้ที่เดียวจบ ไม่เหมือนเก็บใน Excel ที่ต้องไล่แก้ทีละแถว

## ปฏิภาณ พิพัฒน์พิทยากุล — Key-Value

ผมรับผิดชอบ Key-Value Database ครับ ศึกษาว่าข้อมูลถูกเก็บแบบคู่ Key กับ Value เช่น menu:M001:price แล้วเปรียบเทียบกับ Relational ว่า Key-Value ต้องรู้ Key แน่นอนล่วงหน้าถึงจะดึงข้อมูลได้ ค้นหาแบบมีเงื่อนไขไม่ได้ ผมสรุปว่าเหมาะกับข้อมูลที่ต้องอ่านเร็วมาก ๆ อย่างราคาที่โชว์หน้าจอ POS หรือใช้เป็น Cache

## ณัฐพงศ์ เหินหาว — Document

ผมรับผิดชอบ Document Database ครับ ดูว่าข้อมูลเมนูถูกเก็บเป็น JSON ก้อนเดียว รวมวัตถุดิบไว้ในนั้นเลย ข้อดีคืออ่านเร็วและยืดหยุ่น เพิ่ม field พิเศษได้โดยไม่กระทบเมนูอื่น แต่ข้อเสียคือถ้าเปลี่ยนชื่อเมนู ต้องไล่แก้ทุกที่ที่ชื่อซ้ำอยู่ ไม่งั้นข้อมูลจะไม่ตรงกัน ซึ่งเชื่อมกับปัญหา Data Redundancy ที่เรียนในบทที่ 1

## [คนที่ 4] — Graph

ผมรับผิดชอบ Graph Database ครับ ดูว่าข้อมูลเก็บเป็นโหนดกับเส้นเชื่อมความสัมพันธ์ เช่น สมาชิกกับเมนู เชื่อมด้วย ORDERED, LIKES, REFERRED จุดเด่นคือตอบคำถามแบบ "ใครชอบอะไร มักสั่งอะไรอีก" ได้เร็วกว่า Relational มาก เพราะไม่ต้อง JOIN หลายตาราง

---

## คำถามที่อาจารย์อาจถามต่อ

**"ทำไมไม่ใช้แบบเดียวทั้งระบบ"**
แต่ละแบบเก่งคนละด้าน ระบบใหญ่จริงมักผสมหลายแบบ (Polyglot Persistence)

**"ยกตัวอย่างจากร้านกาแฟที่จำได้"**
- Relational: ออร์เดอร์ O001 สั่งทั้งอเมริกาโน่และลาเต้ ต้องแยก ORDER_ITEM
- Key-Value: GET menu:M001:price ต้องรู้ Key ล่วงหน้า
- Document: ลาเต้มี field seasonal_note ที่เมนูอื่นไม่มี
- Graph: สมชายชอบลาเต้ → เดินตาม Edge LIKES หาเมนูที่คล้ายกัน
