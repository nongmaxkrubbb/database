## 📝 สคริปต์เต็ม (พูดตามได้เลย)

### [0:00–0:30] เปิดกรณีศึกษา — คนที่ 1

> "สวัสดีครับ/ค่ะ กลุ่มของเราจะพาไปดูร้านกาแฟ 'ดอยเด่น' ที่มีข้อมูลต้องจัดเก็บ 3 ส่วน คือ เมนู 3 อย่าง — อเมริกาโน่ ลาเต้ ช็อกโกแลต, สมาชิก 2 คน — สมชายกับมาลี ที่มีแต้มสะสม, และคำสั่งซื้อวันนี้ 2 รายการ
>
> คำถามที่เราตั้งไว้คือ ข้อมูลชุดนี้ควรเก็บในฐานข้อมูลแบบไหน? วันนี้เราจะพาไปดูว่า 4 รูปแบบหลัก ๆ ตอบคำถามนี้ต่างกันยังไง เริ่มจาก Relational ก่อนเลยครับ/ค่ะ"

### [0:30–0:50] Relational Model — คนที่ 1 (ต่อ)

> "Relational เก็บข้อมูลเป็นตาราง เหมือน Excel เราแบ่งข้อมูลร้านกาแฟออกเป็น 5 ตาราง คือ MENU, INGREDIENT, MEMBER, ORDER และ ORDER_ITEM
>
> จุดที่น่าสนใจคือทำไมต้องแยก ORDER_ITEM ออกจาก ORDER — เพราะ 1 ออร์เดอร์มีได้หลายเมนู เช่น สมชายสั่งทั้งอเมริกาโน่และลาเต้ในออร์เดอร์เดียว ถ้าไม่แยกตารางจะเก็บข้อมูลซ้ำเยอะมาก และถ้าจะแก้ราคาลาเต้ เราแก้แค่ 1 แถวในตาราง MENU เท่านั้น ไม่ต้องไปไล่แก้ทุกที่"

### [0:50–1:10] Key-Value Database — คนที่ 2

> "ต่อกันด้วย Key-Value ครับ/ค่ะ แบบนี้เก็บข้อมูลเหมือนพจนานุกรม ทุกอย่างมี Key กับ Value เช่น `menu:M001:price` แล้วได้ค่า 60 กลับมา
>
> ข้อดีคือเร็วมาก แต่ข้อจำกัดคือต้องรู้ Key ที่แน่นอนล่วงหน้า จะค้นแบบ 'หาเมนูราคาต่ำกว่า 65 บาท' แบบ Relational ทำไม่ได้ เพราะงั้น Key-Value จะเหมาะกับข้อมูลที่ต้องอ่านบ่อยและรู้ Key อยู่แล้ว เช่น ราคาปัจจุบันของเมนูที่โชว์หน้าจอ POS"

### [1:10–1:30] Document Database — คนที่ 3

> "ของผม/หนูคือ Document Database ครับ/ค่ะ เก็บข้อมูลเป็น JSON เหมือนแฟ้มเอกสาร 1 ฉบับ อย่างเมนูอเมริกาโน่ก็เก็บชื่อ ราคา วัตถุดิบไว้ในก้อนเดียวกันเลย ไม่ต้องแยกตาราง
>
> ข้อดีคืออ่านเร็ว ได้ครบในครั้งเดียว และยืดหยุ่นมาก อย่างลาเต้มี field พิเศษ 'seasonal_note' ที่เมนูอื่นไม่มีก็ใส่ได้เลย แต่ข้อเสียคือถ้าเปลี่ยนชื่อเมนู ต้องไปแก้ทุกที่ที่ชื่อนี้ถูกเก็บซ้ำไว้ เช่น ในออร์เดอร์เก่า ไม่งั้นข้อมูลจะไม่ตรงกัน"

### [1:30–1:50] Graph Database — คนที่ 4

> "สุดท้าย Graph Database ครับ/ค่ะ เก็บข้อมูลเป็นโหนดกับเส้นเชื่อมความสัมพันธ์ ในร้านกาแฟ โหนดคือสมาชิกกับเมนู ส่วนเส้นเชื่อมคือความสัมพันธ์ เช่น ใครสั่งอะไร ใครชอบอะไร ใครแนะนำใครมาสมัคร
>
> จุดแข็งของ Graph คือตอบคำถามแบบ 'ลูกค้าที่ชอบลาเต้ มักสั่งอะไรอีก' ได้เร็วมาก เพราะแค่เดินตามเส้นเชื่อม ในขณะที่ Relational ต้อง JOIN หลายตารางซับซ้อนกว่ามาก เหมาะกับระบบแนะนำสินค้าโดยเฉพาะ"

### [1:50–2:15] สรุปว่าใช้โมเดลไหนกับข้อมูลส่วนไหน — คนที่ 1 (หรือรวมกันพูด)

> "สรุปแล้วกลุ่มเรามองว่า ข้อมูลเมนูกับสมาชิก เหมาะกับ Relational เพราะโครงสร้างชัดเจน ต้องการความถูกต้องสูง ราคาที่เปลี่ยนบ่อยแบบ Real-time เหมาะกับ Key-Value เพราะต้องการความเร็ว ส่วนระบบแนะนำเมนูที่ต้องดูว่าใครชอบอะไร เหมาะกับ Graph ที่สุด"

### [2:15–2:40] สิ่งที่แปลกใจ — คนที่ 2 หรือ 3

> "สิ่งที่กลุ่มเราแปลกใจที่สุดคือ ข้อมูลชุดเดียวกันสามารถเก็บได้ถูกต้องในทั้ง 4 รูปแบบเลย แต่ไม่มีรูปแบบไหนดีที่สุดตายตัว ขึ้นอยู่กับว่าโจทย์จริงคืออะไร และระบบใหญ่ ๆ ในชีวิตจริงก็มักจะผสมใช้หลายแบบพร้อมกัน ไม่ได้ใช้แบบเดียวทั้งระบบ"

### [2:40–3:00] คำถามที่ยังสงสัย + ปิดท้าย — คนที่ 4 (หรือรวมกัน)

> "คำถามที่กลุ่มเรายังสงสัยคือ ถ้าระบบใช้ทั้ง Relational กับ Key-Value พร้อมกัน เช่น cache ราคาไว้ จะมั่นใจได้ยังไงว่าราคาใน Cache จะไม่ขัดกับฐานข้อมูลหลักเวลามีการเปลี่ยนราคา ต้องมีกลไกอะไรมาช่วยซิงค์ข้อมูลระหว่างสองระบบนี้
>
> ขอบคุณครับ/ค่ะ กลุ่มของเรายินดีตอบคำถามครับ/ค่ะ"
